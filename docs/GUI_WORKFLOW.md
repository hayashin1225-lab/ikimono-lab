# GUI開発ワークフロー

## 役割

### 人間

製品ビジョン、感覚的な違和感、案への直感的評価、製品らしさの判断、最終承認、公開版の実利用確認を担う。人間へGUI仕様書の記述を要求しない。

### ChatGPT：GUIデザイン会議

原発話を保存し、問題抽出、複数解釈、UX要求化、GUI原則との照合、情報優先順位、改善案比較、実装難易度・リスク、Figma検証内容、Codex向けPR分割、実装後の企画・UX監査を担う。

### Figma設計担当

現在画面の記録、比較案、承認案固定、情報階層の視覚化、スマートフォン・PC設計、状態別画面、再利用部品、実装スクリーンショットとの比較を担う。ゲームルールの独自変更、新機能追加、製品目的変更、未確定事項の独断確定、実ブラウザ固有挙動の代替は担当しない。

### Codex

現行コード調査、承認済みFigmaと仕様の確認、最小差分実装、レスポンシブ対応、検査、commit・push・PR作成、未実装事項と不明点の報告を担う。未確定の視覚判断を独断で決めない。

### GitHubと公開環境

GitHubはIssue・仕様・PR・判断履歴を保持する。公開環境は実ブラウザ、実端末、実利用評価の対象である。

## 標準フェーズ

1. **フェーズA：感覚入力** — 人間が音声、文章、スクリーンショットで自由に入力する。
2. **フェーズB：GUIデザイン会議** — 原発話、観察された事実、解釈候補、UX上の問題、維持すべき要素、改善仮説、未確定事項、人間が判断する事項を分離する。
3. **フェーズC：Figma比較案** — 必要な場合に比較案を作成する。基準はスマートフォン375pxとPC1280px、必要に応じて320px、390px、430px、短い画面高を加える。
4. **フェーズD：人間による視覚承認** — 自然な言葉による評価を許容し、ChatGPTが設計判断へ変換する。
5. **フェーズE：実装仕様化** — 承認済みFigmaとUX要求をCodex向け仕様へ変換する。
6. **フェーズF：Codex実装** — 最小差分で実装する。
7. **フェーズG：監査** — コード監査、仕様監査、視覚監査、実利用シナリオ確認を別々に報告する。
8. **フェーズH：公開後確認** — 公開URLで配信版を確認する。人間確認前に完全完了としない。

## GUIデザイン会議の標準出力

1. 原発話
2. 対象画面
3. 対象操作・利用シナリオ
4. 観察された事実
5. 観察された問題
6. 解釈候補
7. 利用者への影響
8. UX要求
9. 維持すべき要素
10. 改善案
11. 各案の利点
12. 各案の欠点
13. 実装難易度
14. リスク
15. 推奨案
16. Figmaで確認すべき状態
17. 実装前に人間が判断する事項
18. Codexへ渡す際のPR分割案

原発話を消さず、後から解釈のずれを確認できるようにする。

## 状態遷移

| 状態 | 意味 |
| --- | --- |
| CAPTURED | 原発話・観察を受け取った。 |
| INTERPRETING | 事実・解釈・要求を整理中。 |
| FIGMA_REQUIRED | Figma比較が必要と判断した。 |
| FIGMA_PROPOSED | 比較案または承認候補がある。 |
| HUMAN_REVIEW | 人間の判断待ち。 |
| APPROVED | 方針が承認された。 |
| SPEC_READY | Codex向け仕様が準備済み。 |
| IMPLEMENTING | 実装中。 |
| PR_OPEN | PRでレビュー中。 |
| AUDITING | コード・仕様・視覚・シナリオを監査中。 |
| MERGED | mainへマージ済み。 |
| PUBLISHED | 公開版の配信を確認する段階。 |
| HUMAN_CONFIRMED | 人間が実利用上の評価をした。 |
| CLOSED | 案件を終了した。 |

Figma不要の軽微変更は `INTERPRETING → APPROVED → SPEC_READY` と進められる。ただし実装後監査、対象状態への実ブラウザ到達、公開後確認、人間の最終評価は省略しない。差し戻し例は `HUMAN_REVIEW → INTERPRETING`、`AUDITING → IMPLEMENTING`、`PUBLISHED → IMPLEMENTING`、`HUMAN_CONFIRMED → INTERPRETING`。

## Figmaを使う判断基準

### Figma必須

- 情報階層、主要操作の位置、画面構造が変わる。
- 新しい画面、複数比較案、PCとスマートフォンで異なる構造を扱う。
- 固定操作・タブ・モーダルなど複数状態を持つ。
- 複数PRに分けるGUI改修、GUI共通原則への影響、言葉だけで完成像を共有しにくい場合。

### Figma省略可能

- 明確な誤字、単純な文言変更、原因が確定したブラウザ固有バグ。
- 既存承認済みコンポーネントの再利用、数px程度の明確な崩れ、Figmaとの差異を戻すだけ、新しいGUI判断を伴わない回帰修正。

Figmaを省略しても実利用シナリオ確認は省略しない。

## Figmaファイルの推奨初期構造

| ページ | 責務 |
| --- | --- |
| 00_GUI_CHART | GUIカルテ。GUI共通原則、現在の主要課題、過去の重要判断、成功例、失敗例、既知のブラウザ問題、確認対象端末、現在改善中の画面を置く。 |
| 01_CURRENT | 現在実装されている画面を置く。可能なら実装スクリーンショットを使い、設計上の再現図は実画面と混同しない表示にする。 |
| 02_PROPOSALS | 比較中の案を置く。各案に案名、狙い、解決する問題、利点、欠点、実装負荷、リスク、未確定事項を付ける。 |
| 03_APPROVED | 人間が承認した案だけを置く。対象Issue、承認状態、承認日、対象状態、Codexが参照するフレームを記録する。 |
| 04_COMPONENTS | 明確に再利用する部品だけを置く。初期候補は主ボタン、副ボタン、生き物カード、研究ステータス、パック、モーダル、固定操作領域、タブ、バッジ。 |
| 05_REVIEW | 承認済みFigma、実装スクリーンショット、差異、許容差異、要修正、実ブラウザでのみ発生した問題、公開後に発見された問題を置く。 |
| 99_ARCHIVE | 不採用案、旧デザイン、再検討可能な履歴を削除せず保存する。 |

最初から全面的なデザインシステムを構築しない。

## 監査結果の確認用URL

- PR作成後の監査では、PR確認URLを提示する。
- マージ後の監査では、公開URLを提示する。
- GUI変更では、必要に応じてマージコミットSHA等を付けたキャッシュ回避URLを提示する。
- 未公開で確認URLを作れない場合は、その理由を明記する。
- 「監査完了」と「公開版実機確認完了」を分けて報告する。

## GUI案件の完了条件

### 要求

- 原発話、対象画面・シナリオ、観察事実と解釈、UX要求、維持要素が記録されている。

### 設計

- 採用案と不採用案・理由が明示され、必要時はFigma承認画面、情報階層、状態別表示、レスポンシブ規則が定義されている。

### 実装

- Codexが承認済み仕様を参照し、最小差分で実装し、対象外差分がなく、Figmaとの差異を記録している。

### 監査

- コード・仕様・視覚監査、実利用シナリオ確認、実操作での対象状態到達が完了し、未実施項目が明記されている。

### 公開後

- 公開URL（必要時はキャッシュ回避URL）が提示され、公開版で対象シナリオを確認し、人間が最終的な感覚評価を行い、原則・失敗例・判断履歴を更新する。

コードが正常に動くだけでは完了としない。

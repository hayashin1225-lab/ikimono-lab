# アーキテクチャ

## 現在のアプリ構成

- `index.html` 単一ファイルにHTML、CSS、JavaScriptを内包する。
- 画像はdata URI、サーバー・外部ライブラリはなし、GitHub Pagesで公開する。
- 進行は `localStorage` に保存し、iPhone Safariを主要対象とする。

## 段階1半自動開発体制

- **人間**：感覚・違和感・最終判断。
- **ChatGPT**：原発話保存、UX要求、仕様化、工程管理。
- **Figma**：比較案、承認済み画面、視覚上の正本。
- **Codex**：現行調査、最小差分実装、検査。
- **GitHub**：Issue、仕様、PR、判断履歴。
- **公開環境**：実ブラウザ・実端末・実利用評価。

## 成果物の流れ

原発話 → Issue → UX要求 → GUI仕様 → Figma承認画面 → Codex実装 → Pull Request → 公開環境 → 実利用評価 → 原則・判断履歴への反映。必要に応じて差し戻し・反復する。

## Figmaの位置づけ

Figmaはコード、製品目的、状態遷移の完全な正本ではない。視覚配置・寸法・情報階層の承認面であり、実装と比較する共有作業面である。

## 差異の分類

Figmaとコードの差異は、自動的にどちらかを正しいとしない。Codexの不要な逸脱、技術的に必要な差異、Figma側の不足、実利用による改善を区別し、判断理由を記録する。

## 将来の複数AIオーケストレーション

この節は将来構想である。現在、複数の有料AIが自動連携していること、特定の追加AIを実運用していることを示すものではない。

### 論理的な役割と製品の分離

現在は便宜上、ChatGPT、Figma、Codex、GitHubという製品名を用いている。しかし開発体制上の論理的な役割と、その時点で使用する特定製品・モデルは同一ではない。たとえば現在ChatGPTが担う主幹・要求整理の役割を、将来も同じ製品が常に担うとは限らない。

品質、費用、機密性、可用性、得意分野に応じて、役割を担うAIやサービスを変更できる構造を目指す。工程全体を特定製品へ固定しない。

### 将来追加し得る論理的な役割

必要に応じて、主幹・工程統合、要求整理、GUI設計、実装、独立査読、調査・事実確認、テスト設計、回帰検査、セキュリティ監査、法務・規約・コンプライアンス支援、運用監視、障害分析を追加できるものとする。

Claude等の異なる系統の有料AIを使う可能性はあるが、これは将来の候補・例であり未採用である。実運用前には品質、費用、機密性、可用性を評価する必要がある。特定製品の採用を現在確定した事項として扱わない。

### 異なるAIを追加する目的

複数AIの目的は、同じ成果物を無条件に複数AIへ作らせることではない。各AIへ異なる責任と検査観点を割り当てる。

- 主幹AIは原要求を整理し、工程を統合する。
- 実装AIは承認済み仕様をコードへ反映する。
- 独立査読AIは主幹AIの前提や仕様の欠落を疑う。
- テストAIは利用シナリオと回帰条件を設計する。
- セキュリティAIは保存、入力、依存関係、公開範囲を検査する。
- GUI監査AIは実装せず、視覚上の問題を報告する。

同じAIが要求整理、実装指示、監査まで担当すると、自ら設定した前提を十分に疑えない場合がある。異なる系統のAIを独立査読へ配置する主な目的は単純な能力の加算ではなく、誤りの相関を下げることである。複数AIの意見が一致したことだけを正しさの根拠にはしない。

### 重複実行の原則

同一成果物を複数AIが無条件に重複作成する運用は標準としない。重複実行は、重要な設計判断、高リスクな変更、セキュリティまたは個人情報、保存互換性、決済や商用契約、公開後の影響が大きい変更、主幹AIの判断に不確実性がある場合、独立した反証が必要な場合に限定する。

軽微な文言修正や明確な回帰修正まで常に複数の高額AIへ渡す必要はない。品質向上とAI利用費の均衡を取る。

### モデル選択

将来は、要求される品質、作業の重要度、誤りが生じた場合の影響、機密性、入力可能なデータ範囲、コンテキスト量、コード・文章・GUI・調査等の得意分野、応答速度、利用費、サービスの可用性、ベンダー障害時の代替可能性を考慮して担当AIを選択できる構造を目指す。最高性能または最高価格のAIをすべての作業へ一律に使用しない。

### 共通成果物による連携

複数AIは互いの会話履歴や記憶だけを前提に引き継がない。GitHub Issue、原発話、仕様書、設計判断、承認済みFigma、コード差分、Pull Request、テスト結果、未確定事項、公開URL、実利用評価、GUIカルテ、判断履歴を共有成果物として連携する。

各AIが何を入力として読み、何を出力し、どの役割へ渡したかを追跡可能にする。

### 権限分離

将来AIが増えても、すべてのAIへ同じ権限を与えない。査読AIはコードを直接変更せず、実装AIは製品目的を変更せず、GUI AIはゲームルールを変更せず、調査AIは公開を承認せず、テストAIは受入条件を独断で緩和しない。どのAIも `main` へ自動マージせず、人間の最終判断を代替しない。

役割、読み取り対象、書き込み対象、承認権限を分離できる構造を目指す。

### 人間の役割

自動化が進んでも、製品ビジョン、商用化の判断、製品らしさ、利用者へ提供してよいか、法的・倫理的に許容できるか、AI利用費と品質の均衡、重要な仕様変更、最終公開承認、AI同士の意見が分かれた場合の判断は人間へ残す。AIによる合議結果を人間の承認なしに最終決定としない。

### 段階的な導入

現在の段階1は、人間、ChatGPT、Figma、Codex、GitHub、公開環境による最小構成である。将来の複数AI導入は必要性と効果を確認しながら段階的に行う。

| 段階 | 将来構想の例 |
| --- | --- |
| 段階1 | 主幹AI、GUI共有面、実装AI、GitHubによる証跡、人間の承認。 |
| 段階2 | 独立査読AI、テスト設計AI、調査AI。 |
| 段階3 | セキュリティ監査、法務・コンプライアンス支援、運用監視、案件ごとのモデル振り分け。 |

この段階区分は暫定構想であり、現在の実装済み機能とは区別する。

### 評価指標

複数AI導入の効果は、文章の自然さや回答のもっともらしさだけで評価しない。将来的には、仕様逸脱率、手戻り率、公開後不具合率、GUI不具合検出率、回帰不具合率、人間の確認時間、1案件あたりの処理時間、1案件あたりのAI利用費、独立査読による検出件数、誤検出率、未実施検査の発生率、障害時の復旧時間を評価候補とする。

複数AIを追加したことで、実際に品質向上または人間負荷低減が得られたかを検証する。

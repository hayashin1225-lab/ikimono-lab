# いきもの研究所 リードコーディング開発運用規約 v1.0（正本）

- 文書種別：開発運用規約・正本
- 適用対象：プロダクトオーナー、ChatGPT GM、Codex、連絡将校、GitHub上のAI駆動開発
- 制定日：2026-07-21
- 関連正本：`docs/governance/model-selection-and-escalation-policy.md`

## 1. 目的

本規約は、ChatGPT GMが開発の主線を担う「リードコーディング」方式を定める。

Codexを要件理解、設計、実装、デバッグ、報告の全工程へ常時投入するのではなく、ChatGPT GMが要件、設計、コード、テスト設計、GitHub上の変更管理を主導し、Codexを実行環境検証、独立査読、高難度デバッグ、最終統合検証へ集中させる。

目的は、次の両立である。

- Codex計算資源の節減
- 人間によるAI間の逐次中継の削減
- 実装速度の向上
- 独立査読による品質維持
- GitHub上の変更履歴と承認境界の維持

## 2. リードコーディングの定義

リードコーディングとは、ChatGPT GMが次を一貫して主導する開発方式である。

- 要求の確認とSPDD化
- 制約と変更禁止範囲の固定
- 既存コードの読解
- 設計判断
- 実装コードの作成
- 単体試験、契約試験、E2Eの設計
- GitHub Issue、branch、commit、Pull Requestの管理
- Codexへ渡す検証対象と完了条件の明確化
- Codexの査読結果に基づく修正判断
- 所有者へ提示する承認材料の整備

Codexは同じ工程を無条件に最初からやり直すのではなく、ChatGPT GMが形成した実装を別の処理系として疑い、壊し、実環境で検証し、必要な場合に高度化する。

## 3. 役割分担

### 3.1 プロダクトオーナー

プロダクトオーナーは、目的、優先順位、受入判断、人間承認境界を保持する。

責任範囲：

- 事業目的とユーザー価値の決定
- 要求と優先順位の承認
- 仕様上の重要な選択
- merge、公開、課金、破壊的操作の承認
- 最終的な受入または差戻し

プロダクトオーナーは、通常の構文エラー、fixture不足、mock不足、テスト失敗の逐次デバッグ中継を担当しない。

### 3.2 ChatGPT GM兼リードコーダー

ChatGPT GMは開発の主担当である。

責任範囲：

- 要件定義とSPDD
- 変更範囲と禁止範囲の固定
- 設計、実装、テスト設計
- GitHub上の正本管理
- Issue、branch、commit、PRの作成と更新
- 差分監査
- Codex向け検証パッケージの作成
- Codexの結果を踏まえた修正または再設計
- 完了条件と未完了条件の判定

リードコーダーは、自身が直接扱える作業をCodexへ丸投げしない。

### 3.3 Codex

Codexは、実行環境に接続された独立検証・高度化担当である。

標準責任範囲：

- 実リポジトリ環境での試験実行
- PowerShell、Git、CLI、複数プロセス等の環境固有検証
- 単体試験、契約試験、E2Eの実行
- 回帰、境界条件、失敗遷移の査読
- セキュリティ、権限、破壊的操作の監査
- 原因再現が難しい障害の調査
- merge候補の最終統合検証
- リードコーダーが明示的に委任した高度な修正

Codexは、依頼された査読に対して無条件に全体を再実装しない。

### 3.4 GitHub

GitHubは正本と実行記録を保持する。

- 仕様正本
- 開発運用規約
- Issue
- commit
- Pull Request
- review
- test report
- 承認履歴

会話ログは思考過程と制定経緯であり、GitHub上の正本を置き換えない。

### 3.5 連絡将校

連絡将校は将来、実行統制を担当する。

- タスク状態の管理
- 実行担当モデルの選定
- 失敗署名と修正ループの追跡
- モデル昇降格
- 人間への逐次デバッグ中継の抑止
- ChatGPT GM、Codex、GitHub間の引継ぎ

連絡将校は意味判断を独占せず、正本と明示された完了条件に従う。

## 4. 標準開発フロー

### フェーズ1：要求固定

ChatGPT GMが次を整理する。

- 原要求
- 欲しい成果
- 周辺事情
- 変更禁止範囲
- 完了条件
- 試験条件
- 承認が必要な操作
- 推奨モデルと推論強度

仕様の重要な不明点だけをプロダクトオーナーへ返す。

### フェーズ2：リード実装

ChatGPT GMが主担当として次を行う。

- 既存コード調査
- 設計
- コード変更
- テスト追加
- 静的確認
- GitHub branch、commit、PRの整備

実行環境が利用できる場合は、実行可能な試験も行う。

### フェーズ3：内部確認

Codex投入前に、ChatGPT GMが次を確認する。

- 差分が要求範囲内である
- protected pathを変更していない
- テストが要求を証明する
- 不要な機能拡張がない
- Codexへ渡す論点が明確である

### フェーズ4：Codex独立査読

Codexには、原則として次の順序で依頼する。

1. 現行差分を実環境で実行する
2. 要求、契約、回帰、失敗遷移を査読する
3. 問題がなければ検証結果だけを報告する
4. 問題がある場合は、原因と最小修正案を提示する
5. 環境依存または高難度の場合のみ、明示された範囲で修正する

Codexへ「全部作り直す」「最初から実装する」という重複依頼を既定としない。

### フェーズ5：修正担当の振分け

査読結果に応じて担当を決める。

- 原因が明確な軽微修正：ChatGPT GM
- 設計または契約の変更：ChatGPT GM
- 実行環境固有の障害：Codex
- 複数プロセス、CLI、Git、OS固有障害：Codex
- 単純反復、整形、確定済み変更：下位モデルまたはChatGPT GM
- 重要な統合障害：上位Codexモデル

### フェーズ6：最終検証と承認

merge候補では次を確認する。

- 完了条件を満たす
- 必須試験が成功する
- 回帰がない
- PRのhead SHAと報告SHAが一致する
- worktreeがcleanである
- 実行禁止操作を行っていない
- 人間承認境界を維持している

mergeはプロダクトオーナーの明示承認後に行う。

## 5. Codexを呼ぶ標準チェックポイント

Codexは細かな修正ごとではなく、意味のあるチェックポイントで呼ぶ。

標準チェックポイント：

- 最初の動作可能版が完成した
- APIまたはCLI契約が固定された
- 状態機械または永続stateが完成した
- 複数プロセスE2Eを実行する
- OSまたは実行環境依存の確認が必要
- 重要な回帰試験を追加した
- リリースまたはmerge候補になった
- ChatGPT GMだけでは再現できない不具合がある

通常は、複数の軽微修正をまとめて一回の査読へ渡す。

## 6. Codexを呼ばない標準ケース

次は、原則としてChatGPT GMまたは下位モデルで処理する。

- 文言変更
- README、Issue、PR本文の整形
- 名前変更
- 明確な一ファイル修正
- 既知パターンの反復
- 仕様書や正本文書の追加
- GitHub上の決定的な文書登録
- 上位モデルが方針を確定済みの局所実装
- 実行環境を必要としない差分監査

ただし、高リスクな認証、権限、決済、削除、公開処理は小規模でも独立査読を行う。

## 7. 二重作業の防止

ChatGPT GMとCodexが同じ仕事を重複して行わないため、依頼時に次を明示する。

- 実装済み部分
- 未実装部分
- 査読だけを求める部分
- 修正を許可する部分
- 変更禁止範囲
- 実行すべき試験
- 停止条件
- commit、push、PR、mergeの可否

Codexへの既定文は次の考え方とする。

「この差分を実環境で実行し、契約不一致、回帰、境界条件を独立査読する。問題がなければ再実装しない。必要な場合のみ、許可範囲で最小修正する。」

## 8. Codexへの引継ぎパッケージ

Codexへ作業を渡す場合、最低限次を含める。

- リポジトリ名
- Issue番号
- PR番号
- base branch
- work branch
- 現在のHEAD
- git status
- 変更ファイル
- 新規ファイル
- 完了済み事項
- 未完了事項
- 失敗した試験と完全なエラー
- 試した修正と結果
- protected path
- 実行すべき試験
- commit、push、PR、mergeの可否
- 推奨モデルと推論強度
- 真の停止条件
- 最終報告形式

未コミット差分が存在する場合は、破棄、reset、clean、checkout上書きを禁止する。

## 9. 査読依頼の標準レベル

### レベルA：軽量確認

対象：小規模で低リスクな差分。

- diff確認
- 構文確認
- 既存テスト実行

### レベルB：独立査読

対象：通常の機能追加、複数ファイル変更。

- 要求との一致
- 回帰
- 境界条件
- テスト妥当性
- 失敗時挙動

### レベルC：統合検証

対象：複数プロセス、Git、CLI、状態機械、永続state、移行。

- 実環境E2E
- 失敗注入
- 状態復元
- remote refs
- clean worktree
- 権限と破壊的操作

### レベルD：高度化・再設計

対象：既存設計では解消できない問題、重大な回帰、契約矛盾。

- 根本原因分析
- 契約再整理
- 状態機械再設計
- 上位モデルでの実装修正

モデルと推論強度は、関連正本 `model-selection-and-escalation-policy.md` に従う。

## 10. 計算資源の運用

計算資源は単純な使用量ではなく、次の総コストで評価する。

- モデル利用量
- 経過時間
- 人間の中継時間
- 同一障害の反復
- 回帰修正
- 引継ぎ損失
- 完了確率

安価なモデルで人間の中継と再試行が増える場合、それは節減ではない。

Codex資源を節減するため、次を標準とする。

- 要件整理と設計はChatGPT GMで先に固定する
- Codexへ完全な引継ぎパッケージを渡す
- 細かな変更をまとめて査読する
- 査読対象をdiff、契約、試験へ限定する
- 問題がなければ再実装させない
- 失敗反復時は同じモデルで粘らず昇格する
- 単純な後工程は降格する

## 11. 人間承認境界

次は、ChatGPT GM、Codex、連絡将校のいずれも自動で越えてはいけない。

- merge
- publish
- deployment
- 課金
- 外部送信
- 本番データ変更
- 破壊的削除
- 権限付与
- セキュリティ設定変更
- protected pathの例外変更

所有者が個別に明示承認した場合だけ実行する。

## 12. 真の停止条件

次の場合だけプロダクトオーナーへ判断を求める。

- 仕様が相互矛盾する
- 重要な選択肢に事業判断が必要
- 権限または資格情報が不足する
- 必要な実行環境が存在しない
- 外部サービスが停止する
- 承認境界に到達する
- 保存中の差分が破損し、復元方針が必要

次は、原則としてAI間で解決する。

- 構文エラー
- テスト失敗
- mockまたはfake不足
- JSON不正
- fixture不足
- PATHまたは環境変数不足
- 状態不整合
- 既存コードの調査不足
- モデル能力不足

## 13. PR #14の移行例外

PR #14 `feat: add liaison officer MVP runtime` は、本規約への移行例外とする。

理由：

- Codex環境に未コミット差分が存在する
- fake gh移行途中の生成物とログがある
- 実Gitと一時bare remoteの準備が済んでいる
- セッションをまたいで全面移送すると再構築コストが増える

したがって、PR #14は現在のCodex Sol maxセッションで完遂する。

PR #14完了後に開始する新規Issue・新規PRから、ChatGPT GMによるリードコーディングを標準適用する。

## 14. 標準作業指示欄

各Issueまたは作業指示には、必要に応じて次を記載する。

- リード担当
- Codex投入有無
- Codex査読レベル
- 推奨モデル
- 推奨推論強度
- Codex投入チェックポイント
- Codexに許可する変更範囲
- Codexに禁止する操作
- 人間承認が必要な操作
- 最終報告形式

例：

- リード担当：ChatGPT GM
- Codex投入：最初の動作可能版完成後
- 査読レベル：C 統合検証
- 推奨モデル：Sol
- 推奨推論強度：high
- Codexの役割：実環境E2E、失敗注入、回帰査読
- Codexによる再実装：原則禁止
- 修正許可：環境依存障害の最小修正のみ
- merge：所有者承認後

## 15. 成果評価指標

本方式の有効性を次で評価する。

- Issueから初回動作版までの時間
- Codex投入回数
- Codex使用量
- 人間の中継回数
- 同一障害の再発回数
- 初回査読合格率
- merge前の欠陥発見数
- merge後の回帰数
- 引継ぎ失敗数
- 総作業時間

品質を落として使用量だけを削減した状態は成功とみなさない。

## 16. 改定規則

- 本規約はGitHub上の本ファイルを正本とする
- Issue本文、PR本文、会話ログは正本ではない
- モデル名の変更はモデル選定正本側で管理する
- 役割、工程、承認境界の変更は本規約を改定する
- 改定時は版番号、日付、理由を記録する

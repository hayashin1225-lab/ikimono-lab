# AI実装モデル選定・昇降格規則

- 文書種別：正本
- 版：v1.0
- 制定日：2026-07-21
- 制定記録：Issue #15
- 適用対象：ChatGPT GM、連絡将校、Codex、その他の実装エージェント
- 目的：作業難度に応じてモデルと推論強度を選び、失敗時は人間をデバッグ中継にせず、自動的に昇格・分解・再実行する。

## 1. 正本性

本ファイルを、いきもの研究所におけるAI実装モデル選定・昇降格規則の正本とする。

Issue、Pull Request、会話ログ、作業報告は、制定経緯または実行記録として扱い、本規則本文の正本とはしない。

本規則を変更する場合は、Pull Requestによる差分確認と所有者の明示承認を必要とする。

## 2. 基本原則

1. モデルは常に最上位を使用するのではなく、成功確率、人間介入コスト、実行コストの合計で選ぶ。
2. 実装不足、fixture不足、構文エラー、テスト失敗、環境変数漏れ、fake／mock実装不足は、原則として人間へ返す理由にしない。
3. 同じ種類の失敗を繰り返した場合、同一モデルへ同じ指示を重ねず、モデル昇格または作業分解を行う。
4. 昇格前の差分、ログ、実行コマンド、テスト結果、未完了事項を保存し、上位モデルへ引き継ぐ。
5. 降格は、仕様と契約が固定され、作業が局所的・機械的になった後に行う。
6. merge、公開、課金、破壊的変更、権限付与などの人間承認境界は、モデル昇格によって解除しない。
7. 安価なモデルの利用によって人間の中継が増えた場合、それはコスト削減ではなく総コスト増加として扱う。

## 3. 能力階層と現行モデル割当

モデル名の変更に耐えられるよう、恒久的な能力階層をL1からL3で定義する。

### 3.1 L1：高速・廉価モデル

現行推奨モデル：Luna

推奨用途：

- 文言修正、整形、命名変更
- 既知パターンの反復
- 1ファイル内の小規模で決定的な修正
- 既に通っている手順の再実行
- レポート作成、ログ要約、定型コメント
- 上位モデルが確定した設計に基づく単純実装

非推奨：

- 原因不明のデバッグ
- 複数プロセス、永続状態、Git操作を含むE2E
- 仕様解釈や設計判断が残る作業

標準推論強度：lowからmedium

### 3.2 L2：標準モデル

現行推奨モデル：Terra

推奨用途：

- 通常の実装作業の既定モデル
- 要件と完了条件が明確な機能追加
- 数ファイル程度の変更
- 既存テストに沿った修正
- 単一プロセス内の状態管理
- 原因が特定済みのバグ修正
- 明確なAPI契約に基づくfake／mock実装

非推奨：

- 長時間の自律デバッグ
- 契約自体の逆算が必要な作業
- 複数の外部ツール、プロセス、状態機械を同時に扱う作業
- 同一障害を既に繰り返している作業

標準推論強度：medium

難度が高めであっても契約が明確な場合：high

### 3.3 L3：最上位モデル

現行推奨モデル：Sol

推奨用途：

- 複雑な設計、監査、再設計
- 複数ファイル、複数プロセス、複数ツールをまたぐ実装
- E2E、状態機械、永続state、Git／GitHub連携
- 原因不明、再現条件不明、複数仮説がある障害
- 既存コードから契約を逆算する作業
- 長時間の自律修復
- Terraで失敗が反復した作業
- 失敗時の人間介入コストが高い作業
- リリース前、移行、重要な回帰防止

標準推論強度：high

複数回失敗済み、または重要度が高い場合：max

## 4. 開始時モデル選定規則

### 4.1 最初からL3を指定する条件

次のいずれかに該当する場合、最初からSolを指定する。

- 複数プロセスを起動する
- 外部CLIを2種類以上組み合わせる
- 永続stateまたは状態遷移を扱う
- Git branch、commit、push、Pull Request操作をE2Eで検証する
- fake、mock、wrapperと実コマンドを混在させる
- 既存コードから必要API契約を調査する
- 完了まで自律的な修正と再実行が必要
- 失敗すると人間が何度も中継する可能性が高い
- 破壊的変更、移行、公開前検証など、失敗コストが高い

### 4.2 L2を既定とする条件

次の条件をすべて満たす場合、Terraを既定とする。

- 完了条件が明確
- 主な変更範囲が数ファイル以内
- 使用APIまたはコマンドの契約が既知
- テスト方法が既知
- 障害原因が特定済み、または通常の局所デバッグで解決可能

### 4.3 L1を選択できる条件

次の条件をすべて満たす場合、Lunaを選択してよい。

- 設計判断が残っていない
- 作業が機械的
- 変更範囲が局所的
- 失敗時の影響が小さい
- 上位モデルが作成した手順またはパッチ方針がある

## 5. 実行中の昇格規則

### 5.1 即時昇格

次のいずれかが起きた場合、次の実行から一段階昇格する。

- 同一の根本原因または同一エラーメッセージが2回発生した
- 修正したコードではなく、古い生成物または別経路が実行される問題が再発した
- 「未実装」「追加実装で解決可能」を継続不能として返した
- 人間の判断が不要と認めながら停止した
- 2回以上の修正によって進捗地点が後退した
- 部分修正によって以前通っていた試験が再び通らなくなった
- 作業契約の棚卸しまたは設計変更が必要になった
- 中間エラーを人間へ逐次返し始めた

標準昇格：

- LunaからTerra
- TerraからSol high
- Sol highからSol max

### 5.2 強制昇格

次の場合、現在のモデルでの追加試行を禁止し、Sol high以上へ移す。

- Terraで同種エラーが2回発生した
- Terraで修正ループが3回に達した
- 状態機械、別プロセス、永続stateの不整合が原因となった
- E2Eの不足APIを一件ずつ発見するモグラたたき状態になった
- 人間が2回以上、同じデバッグ指示を中継した

### 5.3 昇格時の引継ぎ必須項目

上位モデルへ次を渡す。

- 目的と完了条件
- 現在のbranch、HEAD、git status
- 変更済みファイル
- 成功済み試験
- 失敗した試験と完全なエラー
- 試した修正と結果
- 実際に実行されたコマンド、PATH、環境変数、生成物
- 変更禁止範囲
- commit、push、mergeの可否
- 人間へ返してよい真の停止条件

前モデルの説明だけではなく、再開可能な実行状態を引き継ぐ。

## 6. 降格規則

次の条件をすべて満たした場合、後続作業を一段階降格してよい。

- 契約試験または主要E2Eが通った
- 未解決の設計判断がない
- 作業を独立した局所タスクへ分割できる
- 各タスクの入力、出力、変更範囲、試験が明示されている
- 直近2工程で回帰がない

降格例：

1. Solがfake ghの全API契約とstate設計を確定する
2. Terraが各コマンドハンドラと契約試験を実装する
3. Lunaがコメント文、README、レポートを整形する

同一障害の修復中には降格しない。主要な成功チェックポイントを通過するまで、昇格後のモデルを維持する。

## 7. モデルの往復防止

- 昇格後は、少なくとも主要成功条件を1つ通過するまで降格しない。
- 降格後に同種エラーが1回でも再発した場合、直前の上位モデルへ戻す。
- 同一タスクで2回昇格した場合、そのタスクは完了までL3から降格しない。
- モデル変更時には指示全文を作り直さず、本正本の完了条件と保存済み状態を継承する。

## 8. 人間へ返してよい停止条件

次だけを「真に継続不能」とする。

- 必要な権限または資格情報がなく、モデル自身では取得できない
- 必要な実行環境または実行ファイルが存在しない
- 外部サービス停止により作業を継続できない
- 仕様が相互矛盾し、どちらを採るか所有者判断が必要
- 破壊的操作、公開、課金、mergeなど、明示承認が必要
- 作業範囲を超える重要な設計変更が必要
- 保存すべき未コミット差分が壊れており、復元方針の判断が必要

次は継続不能理由にしない。

- 構文エラー
- テスト失敗
- fake／mock未実装
- JSON形式不正
- 配列とscalarの不整合
- PATH、環境変数、fixtureの設定漏れ
- 既存コードの調査不足
- 同一モデルの能力不足
- 追加実装で解消可能な不足

## 9. 連絡将校の自動判定用状態

連絡将校は、将来のruntime実装において各作業について最低限次を保持する。

- `current_model`
- `current_effort`
- `failure_signature`
- `repeated_failure_count`
- `repair_loop_count`
- `human_interrupt_count`
- `last_success_checkpoint`
- `changed_files`
- `tests_passed`
- `tests_failed`

判定順序：

1. 開始時ルールで推奨モデルを決定する
2. 実行する
3. 成功した場合、次工程の難度を再評価する
4. 失敗した場合、failure signatureを正規化する
5. 同一signatureが2回なら昇格する
6. repair loopが3回なら昇格する
7. 人間への中間報告を試みた場合、報告を抑止して自律修復を継続する
8. 上位モデルへ状態を引き継いで再実行する
9. 真の停止条件だけ人間へ通知する

```text
if external_blocker or approval_required:
    stop_and_notify_human()
elif same_failure_count >= 2:
    promote_model()
elif repair_loop_count >= 3:
    promote_model()
elif task_became_contract_or_architecture_work:
    promote_to_sol()
elif stable_checkpoint_passed and remaining_work_is_mechanical:
    consider_demoting_one_level()
else:
    continue_self_repair()
```

本節は規則上の要求であり、runtimeへの実装は別Issueで扱う。

## 10. 作業指示書の標準モデル指定欄

各Issueおよび作業指示書には、原則として次を記載する。

```text
推奨モデル：
推奨推論強度：
選定理由：

自動昇格条件：
自動降格条件：

人間へ返してよい停止条件：
人間の承認が必要な操作：
```

記載例：

```text
推奨モデル：Sol
推奨推論強度：high
選定理由：複数プロセス、永続state、fake gh、実Git、一時bare remoteを含む入口E2E

自動昇格条件：同一失敗2回、修正ループ3回、契約再設計が必要
自動降格条件：全API契約試験と初回成功E2E通過後の局所実装

人間へ返してよい停止条件：外部権限、仕様矛盾、破壊的操作承認のみ
人間の承認が必要な操作：merge、公開、課金、破壊的変更
```

## 11. 連絡将校MVPへの現行適用

`relay.entrypoint.tests.ps1`およびfake ghを含む入口E2E作業は、次を正式指定とする。

```text
推奨モデル：Sol
推奨推論強度：high
失敗反復後：Sol max
```

選定理由：

- PowerShell 5.1
- 別PowerShellプロセス
- fake gh／fake Codex
- 実Git
- 一時bare remote
- 永続state
- Issue、label、Pull Request、commentの状態機械
- DryRunとInvoke-Onceの入口E2E

役割分担：

- Sol：`relay.ps1`のghコマンド契約棚卸し、fake gh設計、状態機械、E2E統合、自律デバッグ
- Terra：契約確定後の個別ハンドラ、局所テスト、明確な失敗ケース追加
- Luna：README、試験結果表、Pull Requestレポート、定型コメント、軽微な整形

## 12. 成果評価指標

モデル選定の良否は、単価だけではなく次で評価する。

- 初回完了率
- 自律修復回数
- 同一障害再発回数
- 人間の中継回数
- 完了までの経過時間
- 回帰件数
- 上位モデルへの引継ぎ成功率
- 実行コスト
- 人間時間を含む総コスト

## 13. 改定規則

- モデル名は将来変更されても、L1「高速・廉価」、L2「標準」、L3「最上位」の能力階層を維持する。
- 新モデル追加時は代表タスクで比較し、L1からL3へ割り当てる。
- 同一モデルでも推論強度が異なる場合、強度変更を半段階の昇降格として扱う。
- 規則変更時は版番号、制定日または改定日、変更理由、関連Issueを記録する。
- runtime実装が本規則と矛盾する場合、本規則を優先する。
